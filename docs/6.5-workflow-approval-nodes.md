# Workflow Approval Nodes

## Overview

Approval nodes pause a workflow until one or more people review and approve (or reject) the request. The workflow designer supports five approver types, each determining **who** is eligible to respond.

## Approver Types

### By Permission

Any member who holds an active, unexpired role with the specified permission may approve. This is the most common type for authorization workflows (e.g., "Authorize Rapier Combat").

**Config fields:**
- `approverType`: `"permission"`
- `approverConfig.permission`: The permission name (literal or context variable)

### By Role

Any member who holds an active, unexpired role with the specified name may approve.

**Config fields:**
- `approverType`: `"role"`
- `approverConfig.role`: The role name (literal or context variable)

### Specific Member

A named individual. No permission or role check — the person only needs to be authenticated. Use this for person-to-person approval requests where the approver does not need any special authority.

**Config fields:**
- `approverType`: `"member"`
- `approverConfig.member_id`: The member's ID (literal or context variable)

### Dynamic (from context)

A registered resolver service determines the eligible approver pool at runtime. Used when the approver logic depends on complex business rules.

**Config fields:**
- `approverType`: `"dynamic"`
- `approverConfig.service`: Resolver class name
- `approverConfig.method`: Resolver method name
- Additional config keys defined by the resolver's `configSchema`

### By Policy

Delegates eligibility to a CakePHP authorization policy class. Used when approval rights are tied to entity-level policies.

**Config fields:**
- `approverType`: `"policy"`
- `policyClass`, `policyAction`, `entityTable`, `entityIdKey`

## Context Variable Injection

Approval node configuration supports **context variable references** — values that are resolved at runtime from the workflow's execution context rather than being hardcoded in the workflow definition.

### Syntax

Context variables use the `$.` prefix followed by a dot-separated path:

```
$.trigger.approvalPermission
$.trigger.memberId
$.nodes.some-node.result.someField
```

### How It Works

1. When a workflow is triggered, the **event data** (passed by the controller) is stored directly as the `trigger` context:
   ```php
   // In the controller
   $this->dispatchWorkflowEvent($dispatcher, 'Activities.AuthorizationRequested', [
       'approvalPermission' => $permission->name,
       'memberId' => $member->id,
       // ...
   ]);
   ```

   At runtime, those values become `$.trigger.approvalPermission`, `$.trigger.memberId`, etc.

2. The **approval node** references those context values:
   ```json
   "approverConfig": {
       "permission": "$.trigger.approvalPermission"
   }
   ```

3. At runtime, the workflow engine's `resolveParamValue()` replaces `$.trigger.approvalPermission` with the actual value (e.g., `"Authorize Rapier Combat"`).

### When to Use Context Variables

Use context variables when:
- The **same workflow definition** is used for multiple entity types (e.g., one authorization workflow handles all activity types, each with a different permission)
- The approver depends on data submitted at trigger time
- The required count comes from an app setting (`requiredCount` also supports context/app-setting resolution)

### Designer UI Behavior

The Permission, Role, and Member fields in the approval node config panel include a **mode toggle** (similar to the Required Approvals field):

- **Fixed Value**: Shows an autocomplete search field to select a specific permission, role, or member from the database.
- **Context Variable**: Shows a text input for entering a `$.path` reference that will be resolved at runtime.

When loading a workflow that contains a context variable (e.g., `$.trigger.approvalPermission`), the toggle automatically selects "Context Variable" and displays the path.

The **trigger node** config panel shows a read-only **Trigger Variables** section listing all available `$.trigger.*` paths with their types. These are derived from the trigger event's `payloadSchema` in the registry and are available for reference in downstream node configurations.

### Example: Authorization Workflow

The built-in authorization workflow (`activities-authorization-request.json`) demonstrates this pattern:

```json
{
    "trigger-auth": {
        "type": "trigger",
        "config": {
            "event": "Activities.AuthorizationRequested",
            "entityIdField": "authorizationId"
        }
    },
    "approval-gate": {
        "type": "approval",
        "config": {
            "approverType": "permission",
            "approverConfig": {
                "permission": "$.trigger.approvalPermission"
            },
            "requiredCount": "$.trigger.requiredApprovals",
            "serialPickNext": true,
            "initialApproverId": "$.trigger.approverId"
        }
    }
}
```

The permission name comes from the Activity entity's associated Permission record, injected by the controller's `buildRequestTriggerData()` method at dispatch time.

## Common Configuration Options

These apply to all approver types:

| Option | Type | Description |
|--------|------|-------------|
| `requiredCount` | integer or context ref | Number of approvals needed before the gate passes |
| `allowParallel` | boolean | Whether multiple approvers can act simultaneously |
| `serialPickNext` | boolean | Each approver picks the next one from the eligible pool |
| `deadline` | string | Duration before timeout (e.g., `"7d"`, `"24h"`) |
| `commentWarning` | string | Warning shown above the comment field |
| `initialApproverId` | integer or context ref | Pre-assigned first approver for serial workflows |

## Eligibility Enforcement

Eligibility is enforced at **two levels**:

1. **Listing** (`getPendingApprovalsForMember`): Only shows approvals the member is eligible for, filtering by active permissions/roles.

2. **Submission** (`recordResponse` → `isMemberEligible`): Validates eligibility again at approval time. For permission and role types, checks that the member **still holds** the active permission/role (e.g., warrant has not expired since the approval was listed).

For "Specific Member" type, only the member ID is checked — no permission/role validation is performed, since this type is designed for person-to-person requests.
