# 7.8 Performance Sizing & Risk Monitoring

This guide defines a repeatable benchmark workflow for:

1. Generating installer-facing environment sizing suggestions.
2. Detecting performance risks (slow routes, query-heavy pages, concurrency pressure).
3. Driving targeted optimization work.

## Overview

KMP ships with a performance sizing benchmark script at:

- `/workspaces/KMP/load_test.sh` (wrapper)
- `/workspaces/KMP/app/scripts/perf/sizing-benchmark.js` (implementation)

The benchmark uses authenticated Playwright flows over production-like seed data and can optionally profile route-level SQL workload via MariaDB slow log tables.

## Running the Benchmark

```bash
cd /workspaces/KMP
./load_test.sh
```

Artifacts are written to:

- `test-results/perf/sizing-report-<timestamp>.json`

## Key Environment Variables

| Variable | Default | Purpose |
|---|---|---|
| `KMP_BASE_URL` | `http://127.0.0.1:8080` | Base URL for benchmark traffic |
| `KMP_LOGIN_EMAIL` | `admin@amp.ansteorra.org` | Login identity for authenticated routes |
| `KMP_LOGIN_PASSWORD` | `TestPassword` | Login password |
| `KMP_ROUTE_RUNS` | `5` | Number of sequential profiling runs per route |
| `KMP_CONCURRENCY_LEVELS` | `1,5,10,20` | Virtual-user concurrency levels to test |
| `KMP_SLOW_REQUEST_MS` | `1500` | Slow-route threshold used for risk flags and sizing logic |
| `KMP_CPU_TARGET_UTIL_PCT` | `70` | CPU target utilization used to convert observed host load into capacity guidance |
| `KMP_MEMORY_TARGET_UTIL_PCT` | `80` | Memory target utilization used to convert observed host load into capacity guidance |
| `KMP_TELEMETRY_SAMPLE_MS` | `500` | Host telemetry sample interval (CPU%, memory%, load average) during concurrency tests |
| `KMP_ENABLE_DB_PROFILE` | `1` | Enables per-route SQL query-count/rows profiling |
| `KMP_DB_HOST` | `127.0.0.1` | DB host for SQL profiling |
| `KMP_DB_USER` | `KMPSQLDEV` | DB user for SQL profiling |
| `KMP_DB_PASS` | `P@ssw0rd` | DB password for SQL profiling |
| `KMP_DB_NAME` | `KMP_DEV` | DB name for SQL profiling |
| `KMP_PERF_OUTPUT_DIR` | `test-results/perf` | Report output directory |

Example:

```bash
KMP_CONCURRENCY_LEVELS=1,5,10,20,30 \
KMP_SLOW_REQUEST_MS=1800 \
./load_test.sh
```

## Runtime Instrumentation for Ongoing Monitoring

KMP supports optional request timing logs to identify route regressions in staging/production-like environments.

### Enable request timing instrumentation

Add to `app/config/.env` (or deployment environment):

```bash
PERF_REQUEST_LOG_ENABLED=true
PERF_SLOW_REQUEST_MS=750
PERF_LOG_ALL_REQUESTS=false
PERF_DB_QUERY_LOG_ENABLED=true
PERF_KINGDOM_TAG=ansteorra
```

Optional external sink:

```bash
APPINSIGHTS_CONNECTION_STRING=InstrumentationKey=...;IngestionEndpoint=https://...
APPINSIGHTS_LOG_ENABLED=true
APPINSIGHTS_CLOUD_ROLE=kmp
APPINSIGHTS_LOG_BATCH_SIZE=25
APPINSIGHTS_ERROR_LOG_ENABLED=true
APPINSIGHTS_QUERY_LOG_ENABLED=false
APPINSIGHTS_QUERY_SAMPLE_RATE=10.0
LOG_QUERIES_FILE_SIZE=10MB
LOG_QUERIES_FILE_ROTATE=5
LOG_PERFORMANCE_FILE_SIZE=10MB
LOG_PERFORMANCE_FILE_ROTATE=5
LOG_QUERIES_URL=syslog://your-log-host:514
LOG_PERFORMANCE_URL=syslog://your-log-host:514
```

When enabled, request timing telemetry is emitted to `logs/performance.log` with scope `app.performance` and SQL timing details are emitted through the `cake.database.queries` scope. If `APPINSIGHTS_LOG_ENABLED` is true and `APPINSIGHTS_CONNECTION_STRING` is set, request timing logs and warning/error logs are also sent directly to Application Insights as batched trace telemetry. SQL query export is separately gated by `APPINSIGHTS_QUERY_LOG_ENABLED` and sampled by `APPINSIGHTS_QUERY_SAMPLE_RATE` because query logs can be high volume. Local performance and query files are retained with size-based rotation; tune `LOG_QUERIES_FILE_SIZE`, `LOG_QUERIES_FILE_ROTATE`, `LOG_PERFORMANCE_FILE_SIZE`, and `LOG_PERFORMANCE_FILE_ROTATE` to cap disk use.

### Structured telemetry fields

Every `[request_timing]` event is emitted with the following structured
`customDimensions` (visible in Application Insights, also present in the
local file as `key=value` tokens in the message):

| Field | Source | Use |
|---|---|---|
| `method` | HTTP verb | filter |
| `host` | request host | per-tenant slicing in shared deployments |
| `path` | raw URL path | exact failures / one-off lookups |
| `route_template` | controller/action + `{argN}` | low-cardinality slow-route panels |
| `status` | HTTP status code | error trace pivot |
| `duration_ms` | wall-clock middleware time | p50/p95 latency |
| `memory_peak_mb` | `memory_get_peak_usage(true)` | worker sizing |
| `cpu_user_ms` / `cpu_sys_ms` | `getrusage()` deltas | CPU-bound vs IO-bound triage |
| `query_count` / `db_total_ms` | `RequestQueryCounter` | DB pressure per route |
| `response_bytes` | response body size | bandwidth / cache-miss size |
| `kingdom` | `PERF_KINGDOM_TAG` | multi-kingdom rollup |
| `php_version` / `php_sapi` / `php_os` | runtime fingerprint | platform parity comparison |
| `app_version` | `composer.json` `version` | release-over-release diff |
| `telemetry_schema_version` | `ApplicationInsightsLog::TELEMETRY_SCHEMA_VERSION` | safe forward evolution |

The exporter itself emits one `[telemetry_health]` trace per process at
shutdown (channel `telemetry-health`) summarizing `sent`, `failed`,
`dropped`, `flush_count`, `flush_total_ms`. Use this panel to verify the
overhead of telemetry is not itself distorting the numbers you are
measuring.

### Measurement methodology — DreamHost VPS as trace source

A shared DreamHost VPS is a useful trace source but it is **not** a
benchmark platform. Other tenants on the same host inject noise that
cannot be controlled. Treat measurements from production as relative
trend data, not absolute capacity numbers.

To get reliable numbers from DreamHost data, run three measurement passes
back-to-back and compare:

1. **Baseline**: telemetry off (`PERF_REQUEST_LOG_ENABLED=false`).
2. **Performance only**: `PERF_REQUEST_LOG_ENABLED=true`, `APPINSIGHTS_LOG_ENABLED=true`,
   `APPINSIGHTS_QUERY_LOG_ENABLED=false`.
3. **Performance + sampled queries**: as above with
   `APPINSIGHTS_QUERY_LOG_ENABLED=true` and
   `APPINSIGHTS_QUERY_SAMPLE_RATE=10` (or lower).

The difference in `duration_ms` and `flush_total_ms` between passes is
the telemetry overhead. Under PHP-FPM the exporter calls
`fastcgi_finish_request()` before its HTTPS POST so the response is
already on the wire, but CLI commands (cron, queue workers) still pay
the cost synchronously.

### Scaling to 30 kingdoms — Application Insights cost and retention

Trace volume scales linearly with traffic. Going from 1 kingdom to 30 on
the same workload pattern means roughly 30x ingestion. Before flipping
all kingdoms on, do this:

- Enable the `prod-amp` daily cap (Application Insights → Usage and
  estimated costs) and set it below the workspace's commitment tier
  ceiling.
- Keep `APPINSIGHTS_LOG_BATCH_SIZE=25` (default). Larger batches reduce
  HTTPS overhead but increase per-flush latency.
- Reduce `APPINSIGHTS_QUERY_SAMPLE_RATE` first when the cap is hit. Query
  telemetry has the highest volume and the lowest marginal value once
  the slow-query baselines are established.
- Consider lowering `APPINSIGHTS_QUERY_LOG_ENABLED` to `false` outside
  performance hunts; the local rotated `queries.log` is still available
  on the host if a deep dive is needed.
- Use the **CPU vs wall time** workbook panel to decide whether to scale
  app workers (CPU-bound) or DB (wall-time bound) when adding kingdoms.
- Workspace retention defaults to 30 days. Capture a per-release
  archived export of the `[request_timing]` workbook before retention
  rolls so release-over-release deltas survive.


## How to Use Results for Installer Sizing

The generated report includes:

- Route-level latency summary (avg/p50/p95)
- Concurrency throughput and error rates
- Optional DB profile per route (query count, rows examined, max query time)
- Auto-generated sizing recommendation tier
- Benchmark host telemetry and derived capacity model

### How recommendation values are derived

`recommendation.app` and `recommendation.database` are now derived from measured benchmark-host behavior, not static defaults:

1. Find the highest tested concurrency with:
   - 0% request errors
   - p95 route latency <= `KMP_SLOW_REQUEST_MS`
2. Read host telemetry at that concurrency (CPU p95 and memory p95).
3. Convert observed utilization to capacity using target utilization guards:
   - CPU target: `KMP_CPU_TARGET_UTIL_PCT` (default 70%)
   - Memory target: `KMP_MEMORY_TARGET_UTIL_PCT` (default 80%)
4. Split combined derived capacity into app/db guidance using a documented split model in the report.

This keeps recommendations tied to the actual benchmark machine and observed host pressure.

Use the recommendation as a starting point, then tune for:

- expected concurrent admin users
- plugin mix and workflow complexity
- database latency and I/O characteristics
- mail/queue/background workload

## Performance Risk Triage Workflow

1. Run `./load_test.sh` on current branch and capture report.
2. Prioritize risks in this order:
   - Concurrency errors / high error rate
   - Routes with p95 above threshold
   - DB-heavy routes with high query counts or rows examined
3. For each risky route:
   - Inspect associated grid/contain queries.
   - Validate index coverage and count query strategy.
   - Re-run benchmark after changes.
4. Track trend over time by keeping benchmark artifacts from each release branch.

## Suggested Cadence

- **Before release**: run full benchmark and compare to previous release.
- **After major feature merges**: run targeted benchmark on impacted routes.
- **After schema/index changes**: run DB profile mode to confirm reduced query pressure.
